# LLM Wiki 核心理念

这是一份理念文件，不是一部宪法。

它的用途和 Karpathy 原始 gist 一样：先把模式说清楚，再让 agent 和使用者一起把实现细节实例化出来。默认实现当然重要，但默认实现不应该被误当成唯一正确答案。

## 核心想法

常见的文档型 LLM 工作流更像 RAG：问题来了，模型再去原始文档里找片段、拼答案。这样能工作，但知识不会自然累积。需要综合 5 篇资料的复杂问题，下次再问一次，模型还得重新找一次。

LLM Wiki 的思路不同：

- 原始资料放在 `raw/` 一类的事实层，尽量不改。
- agent 在 ingest 阶段把资料读懂、压缩、联结、更新进 wiki。
- query 优先利用已经整理好的 wiki，而不是每次都从 raw 重建一遍。
- 好的问答、对比、分析和新问题，也可以继续回写进 wiki，让探索本身可复用。

换句话说，wiki 是一个持续演化的知识编译产物，而不是临时检索的副产物。

## 三层结构

### 1. Raw sources

原始资料层。文章、论文、会议记录、图片、数据文件都可以放在这里。它们是事实层，也是最终可回溯的锚点。

### 2. Wiki

由 agent 维护的 Markdown 页面层。这里放摘要、概念、实体、分析、流程、问题，以及这些页面之间的链接。

### 3. Schema

告诉 agent 应该如何维护 wiki 的协议层。它不是产品介绍，而是执行约定：如何 ingest，如何 query，哪些页面需要回写，如何做 lint，如何记录 QA，哪些字段和页面类型在当前仓库里算默认值。

## 关键原则

### 模式优先于写死实现

如果某个目录、字段或页面类型没有明显赚回它的复杂度，就不要硬加。先让结构服务于知识，再让制度服务于结构。

### 页面优先于表格

知识质量来自准确、压缩、可追溯、可导航，不来自把 frontmatter 填得很满。元数据应该帮助阅读和维护，而不是制造“伪精确”。

### 单一真相源

尽量减少需要手工同步的重复表示。

- 页面正文和最小 frontmatter 是真相源。
- `index.md` 适合做导航，但应被当作派生产物。
- `log.md` 负责时间线和审计，不负责定义知识本身。
- 能通过搜索、backlink 或派生逻辑得到的信息，不一定值得再维护一份。

### 写回要有门槛

不是每个聊天结论都应该立刻变成“稳定知识”。

- 临时但可复用的洞察，可以先写为 `draft`
- 经过来源检查、结构整理和必要复核后，再提升为 `stable` 或 `contested`

这样可以防止 wiki 慢慢开始“读自己的总结再生成自己的总结”。

### 自检不等于独立 QA

同一个模型、同一个上下文里的 self-check 仍然有价值，但它只能证明“我做了局部复查”，不能证明“这是独立审阅过的内容”。高风险领域尤其要诚实地区分这一点。

### 规模增长后要换挡

`index.md` 对中小规模 wiki 很好用，但不是永远够用。等到 index 已经不适合在一个上下文里快速扫读，或用户明显开始迷路时，就该加本地搜索层了。可以是 grep，可以是 Obsidian 搜索，也可以是专门的本地 markdown 搜索工具。

## agent 与人的分工

- 人负责找资料、提问题、判断重要性、决定什么时候值得晋升为正式知识。
- agent 负责摘要、整合、交叉引用、重写、维护索引、记录审计。

最理想的状态不是“人替 agent 做整理”，而是“人做判断，agent 做维护”。

## 这份仓库的取向

这个仓库提供的是：

- 一个可直接启动的 prompt
- 一个更轻量的默认 schema profile
- 一个最小样例 vault
- 一个手工回归清单

它不提供的是：

- 所有领域都必须照抄的唯一目录结构
- 一套假装精确的元数据大全
- 一份越写越长、越写越僵硬的超级 prompt
