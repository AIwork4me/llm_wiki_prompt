# 《在AutoClaw中搭建会进化的 Karpathy  LLM Wiki 》

大多数人做知识库，默认思路还是 RAG：资料先堆着，等提问时再检索、再拼答案。Karpathy 在 LLM Wiki 里提出的恰恰是反方向的想法：不要把知识处理放在 query-time，而要尽量前移到 ingest-time。也就是说，LLM 不只是“回答问题的聊天机器”，而是“持续维护 wiki 的知识编译器”。

这件事一旦想明白，知识库的结构就会变。第一层是 `raw/`，只放不可变原始资料；第二层是 `pages/`，放被提炼、压缩、链接、更新后的知识页；第三层是 `SCHEMA.md`，它不是说明文，而是 agent 的执行协议，规定 ingest、query、lint、QA 到底怎么做。真正值钱的不是文件夹，而是这层协议。

在 AutoClaw 里，最容易踩的坑不是“提示词不够长”，而是协议不够硬。我们在实战里最后收敛出几条关键规则：

1. 当前工作目录就是 wiki 的 vault 根目录，不能再套一层 `llm-wiki/`。
2. 模板必须是强制起点，新页和重写页都得从 `_templates/` 出发。
3. `source_ids` 负责来源追溯，`parent_ids / child_ids` 只负责真实层级，不能把 source 页挂成一串“派生子页”。
4. 冲突必须结构化，`status=contested` 时必须有 `conflict_ids`，Lint 也要检查。
5. 没有独立 QA agent 时，不能只写“未执行 QA”，而要落一份 self-check fallback 报告。

这背后的本质，是把知识库从“文档堆”升级成“可进化系统”。一次 ingest 不只是新建几页，而是把新证据编译进旧结构；一次 query 也不只是出答案，而是把有价值的洞察再写回 wiki。这样，知识才会复利。

为什么 AutoClaw 特别适合做这件事？因为它擅长把一串动作串成闭环：先审计目录，再初始化骨架，再生成协议和模板，最后自检 Markdown、补日志、必要时补 QA fallback。换句话说，它不是只帮你“写内容”，而是在帮你建立一个以后还能继续长大的知识工地。

真正好的 LLM Wiki，不是每个页面都写得花哨，而是满足三件事：第一，忠于 raw；第二，页面之间能导航；第三，新资料进来时旧知识能被重写而不是继续堆料。只要这三点成立，你得到的就不是一份 AI 笔记，而是一个会进化的第二大脑。

Karpathy 的启发不在于“让 LLM 帮你记笔记”，而在于把 LLM 变成 wiki 的维护者。对工程团队来说，这比单次回答更重要：因为真正稀缺的不是一次好答案，而是一个会越来越好的知识系统。
